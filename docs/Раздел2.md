
2 Проектирование задачи
2.1 Организация данных

Для разработки базы данных настольного приложения по управлению грузоперевозками была выбрана встраиваемая реляционная СУБД SQLite. Проектирование структуры и управление данными осуществляются с исполь-зованием встроенных средств языка Python (библиотека sqlite3) и утилиты DB Browser for SQLite. В рамках проекта реализована нормализованная модель данных, состоящая из 6 взаимосвязанных таблиц. Логическая структура орга-низована по принципу разделения на справочные данные (водители, города, партнеры, типы грузов), системные данные (пользователи) и операционные данные (журнал перевозок).

2.2 Процессы

Основные функции приложения будут реализованы через взаимодей-ствие пользователя с экранными формами и обработку данных в локальной БД. Система будет поддерживать следующие ключевые бизнес-процессы:
	Авторизация и аутентификация пользователей с разделением ролей (Администратор/Пользователь);
	Управление заявками: создание новой перевозки с выбором маршру-та, груза и исполнителя, редактирование статусов существующих заявок («В пути», «Доставлен»);
	Автоматизация расчетов: математический расчет расстояния между населенными пунктами с учетом коэффициента кривизны дорог и вычисление стоимости рейса на основе тарифа;
	Документооборот: генерация и вывод на печать ТТН на основе дан-ных выбранной заявки;
	Администрирование: управление справочниками городов (с коорди-натами), партнеров и водителей через панель администратора;
	Аналитика: формирование сводного отчета по выручке и количеству заказов в разрезе статусов.

2.3 Описание внешнего пользовательского интерфейса

Пользовательский интерфейс будет представлять собой классическое оконное приложение (Desktop application).
Главное окно разделено на две функциональные зоны:
1	Левая панель (Форма ввода): содержит сгруппированные блоки полей для ввода маршрута, данных о грузе и выбора водителя. Здесь же располага-ются элементы управления (кнопки «Создать», «Обновить», «Очистить»).
2	Правая панель (Таблица данных): отображает список всех перевозок с возможностью сортировки по столбцам и цветовой индикацией статусов.
В верхней части интерфейса (ToolBar) расположены кнопки быстрого доступа к дополнительным модулям: «Управление водителями», «Аналитика», «Экспорт в Excel».
Для взаимодействия с пользователем используются модальные окна: ок-но входа в систему, информационная карточка водителя (досье) с финансовой статистикой, а также окна подтверждения действий (например, при удалении записи). Генерация печатных форм осуществляется через открытие сформиро-ванного документа в браузере по умолчанию.

2.4 Средства защиты и разграничения доступа к данным

В части безопасности приложение обеспечивает защиту доступа за счет обязательной авторизации при запуске. Пароли пользователей не хранятся в открытом виде, а проходят процедуру хеширования (алгоритм SHA-256) пе-ред записью в базу данных.
Доступ к функционалу строго разграничен на основе ролей:
	Пользователь (Диспетчер): имеет право на создание, просмотр и ре-дактирование заявок, печать документов и просмотр информации о водителях. Функция удаления записей и редактирования справочников для него заблоки-рована.
	Администратор: обладает полным доступом, включая удаление за-явок, добавление новых городов и партнеров, а также просмотр финансовой аналитики.
Техническая реализация базы данных SQLite с включенной поддержкой внешних ключей (Foreign Keys) гарантирует целостность данных, предотвра-щая удаление справочных записей, которые используются в активных перевоз-ках.

2.5 Тесты на использование

Для проверки корректности работы системы и удобства пользователь-ского интерфейса будут разработаны функциональные тесты. Тестирование будет проводиться по ключевым сценариям, соответствующим основным вари-антам использования.
В тест-кейсах будут проверены следующие действия:
	Успешный и неуспешный вход в систему;
	Корректность автоматического расчета расстояния и стоимости при выборе городов;
	Создание новой заявки и ее появление в общей таблице;
	Работа фильтрации (поиска) по базе данных;
	Генерация ТТН (соответствие данных в печатной форме данным в таблице);
	Проверка ограничений прав доступа (невозможность удаления записи обычным пользователем).
Все тест-кейсы будут представлены в приложении В.
