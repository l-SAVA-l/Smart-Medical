# Приложение Г. Тест-кейсы для проверки функциональности и БД

Обобщённые тест-кейсы по сценариям, описанным в разделе 2. Проверяется корректность работы приложения и согласованность данных в БД (PostgreSQL, Prisma).

---

## ТК-1. Авторизация и сессии

| Поле | Описание |
|------|-----------|
| **Цель** | Проверить вход, регистрацию и разграничение доступа по ролям |
| **Шаги** | 1) Вход с корректными данными. 2) Вход с неверным паролем / несуществующим пользователем. 3) Регистрация нового пациента. 4) Открытие защищённых страниц без сессии. 5) Открытие админ-панели под ролью Пациент и под ролью Администратор. |
| **Ожидаемый результат** | Успешный вход и редирект в ЛК/админку; при ошибке — сообщение, сессия не создаётся. Регистрация создаёт запись в `patients` с хешем пароля. Без сессии — редирект на страницу входа. Пациент не видит админ-меню; администратор видит полный доступ. |

---

## ТК-2. Запись на приём

| Поле | Описание |
|------|-----------|
| **Цель** | Проверить создание записи и отображение в личном кабинете |
| **Шаги** | 1) Выбор специалиста и (опционально) услуги, даты и времени. 2) Отправка формы записи. 3) Просмотр раздела «Мои записи» в ЛК. 4) Проверка в БД: новая запись в `appointments` с корректными patient_id, specialist_id, service_id, scheduled_at, status. |
| **Ожидаемый результат** | Запись сохраняется, отображается в списке записей пациента с верными данными и статусом. В БД — целостность связей по FK (patient, specialist, service). |

---

## ТК-3. Письма главному врачу

| Поле | Описание |
|------|-----------|
| **Цель** | Проверить создание письма, ответ и переписку, флаги прочтения |
| **Шаги** | 1) Пациент создаёт письмо (тема, текст). 2) Главный врач видит письмо в списке, открывает, отвечает. 3) Пациент видит ответ; добавление сообщений в переписку (letter_messages). 4) Проверка в БД: запись в `letters` (patient_id), в `letter_messages` — цепочка с sender_type; обновление is_read, is_reply_read, has_new_patient_message. |
| **Ожидаемый результат** | Письмо и сообщения сохраняются; флаги прочтения обновляются корректно. Данные в `letters` и `letter_messages` согласованы. |

---

## ТК-4. Чаты с оператором

| Поле | Описание |
|------|-----------|
| **Цель** | Проверить создание чата, назначение оператора, обмен сообщениями и статусы |
| **Шаги** | 1) Пациент создаёт чат. 2) В админ-панели чат отображается со статусом WAITING. 3) Оператор назначает себя на чат. 4) Обмен сообщениями с обеих сторон. 5) Закрытие чата. 6) Проверка в БД: `operator_chats` (patient_id, operator_id, status), `operator_chat_messages` (chat_id, sender_id); счётчики непрочитанных. |
| **Ожидаемый результат** | Чат создаётся, оператор назначается (operator_id заполнен), статусы меняются (WAITING → ACTIVE → CLOSED). Сообщения сохраняются и отображаются; счётчики непрочитанных корректны. |

---

## ТК-5. Редактирование услуги (админ)

| Поле | Описание |
|------|-----------|
| **Цель** | Проверить сохранение изменений услуги в БД и отображение на сайте |
| **Шаги** | 1) Войти под учётной записью с правами на редактирование услуг. 2) Открыть карточку услуги, изменить название, цену или категорию. 3) Сохранить изменения. 4) Проверить отображение на публичной странице услуги. 5) Проверить в БД: запись в `services` обновлена (title, price, category_id, service_category_id). |
| **Ожидаемый результат** | Изменения сохраняются; на сайте отображаются актуальные данные. В БД — обновлённая запись, целостность FK по категориям сохранена. |

---

## ТК-6. Редактирование специалиста и привязка к услугам (админ)

| Поле | Описание |
|------|-----------|
| **Цель** | Проверить сохранение данных специалиста и связь M:N с услугами |
| **Шаги** | 1) Войти под учётной записью с правами на редактирование специалистов. 2) Открыть карточку специалиста, изменить данные (ФИО, специализация, категория). 3) Изменить привязку к услугам (добавить/удалить услуги). 4) Сохранить. 5) Проверить в БД: обновление `specialists`; в `service_specialists` — корректный набор пар (specialist_id, service_id), лишние связи удалены, новые добавлены. |
| **Ожидаемый результат** | Данные специалиста и список его услуг сохраняются и отображаются на сайте. В БД связь M:N согласована. |

## ТК-7. Верификация отзыва (модерация)

| Поле | Описание |
|------|-----------|
| **Цель** | Проверить установку флага верификации отзыва |
| **Шаги** | 1) Войти под учётной записью с правами модерации отзывов. 2) Открыть список отзывов, выбрать отзыв. 3) Установить/снять признак «Верифицирован» (verified). 4) Сохранить. 5) Проверить в БД: в `feedbacks` поле verified обновлено; на сайте отзыв отображается с признаком верификации (если реализовано в интерфейсе). |
| **Ожидаемый результат** | Флаг verified сохраняется в БД и при необходимости отображается в интерфейсе. |

---

## ТК-8. Блокировка пользователя

| Поле | Описание |
|------|-----------|
| **Цель** | Проверить блокировку отправки сообщений пользователем |
| **Шаги** | 1) Войти под учётной записью с правами управления пользователями. 2) Выбрать пользователя (пациента), включить блокировку сообщений (is_messages_blocked = true). 3) Сохранить. 4) Войти под заблокированным пользователем. 5) Попытаться отправить письмо главному врачу или сообщение в чат. 6) Проверить в БД: в `patients` у пользователя is_messages_blocked = true. |
| **Ожидаемый результат** | Заблокированный пользователь не может отправить письмо или сообщение в чат (интерфейс или API возвращают ошибку). В БД флаг is_messages_blocked сохранён. |

---

## ТК-9. Изоляция данных пациента (доступ только к своим данным)

| Поле | Описание |
|------|-----------|
| **Цель** | Проверить, что пациент не может получить чужие письма и чаты |
| **Шаги** | 1) Создать двух пациентов (A и B); от имени B создать письмо и чат. 2) Войти под пациентом A. 3) Попытаться открыть письмо или чат, принадлежащий пациенту B (подмена id в URL или запрос к API с letter_id/chat_id пациента B). 4) Проверить ответ приложения. |
| **Ожидаемый результат** | Пациент A не получает содержимое письма/чата пациента B: ответ 403 Forbidden, 404 или пустой/ошибочный результат. Данные пациента B не отображаются в интерфейсе пациента A. |

---

## ТК-10. Каскадное удаление (чат и сообщения)

| Поле | Описание |
|------|-----------|
| **Цель** | Проверить удаление связанных сообщений при удалении чата |
| **Шаги** | 1) Создать чат с несколькими сообщениями. 2) Зафиксировать id чата и количество записей в `operator_chat_messages` для этого chat_id. 3) Удалить чат (при наличии прав в приложении или через админ-действие). 4) Проверить в БД: запись чата в `operator_chats` удалена; все записи в `operator_chat_messages` с данным chat_id также удалены (Cascade). |
| **Ожидаемый результат** | При удалении чата связанные сообщения удаляются из БД. Нет «висячих» записей в `operator_chat_messages` с несуществующим chat_id. |

---

## ТК-11. Запрет удаления при отсутствии прав

| Поле | Описание |
|------|-----------|
| **Цель** | Проверить, что удаление недоступно ролям без соответствующих прав |
| **Шаги** | 1) Войти под учётной записью без прав на удаление (например, Оператор или Пациент). 2) Попытаться удалить запись (услугу, отзыв, письмо и т.д.) через интерфейс — проверить, что кнопка/действие удаления отсутствует или недоступно. 3) Отправить запрос на удаление напрямую к API (DELETE) с сессией пользователя без прав. 4) Проверить ответ API. |
| **Ожидаемый результат** | В интерфейсе удаление недоступно. API возвращает 403 Forbidden или 401 Unauthorized; запись в БД не удаляется. |

---

**Примечание.** При проведении тестирования в таблицу результатов заносятся: номер тест-кейса, результат (пройден / не пройден), при не пройденном — краткое описание дефекта. Статистика по ошибкам сводится в таблицу 9 раздела 4.
