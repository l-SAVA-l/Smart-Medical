# 2 Проектирование задачи

## 2.1 Организация данных

Для веб-приложения информационного портала медицинской клиники выбрана реляционная СУБД PostgreSQL. Проектирование структуры и работа с данными выполняются с использованием ORM Prisma: схема сущностей описывается в `schema.prisma`, миграции применяются командой `prisma migrate dev`, доступ к данным — через сгенерированный Prisma Client. Реализована нормализованная модель из 18 взаимосвязанных таблиц.

Логическая структура данных организована по принципу разделения на три группы. **Справочные данные:** категории (categories, service_categories с древовидной структурой), услуги (services), специалисты (specialists), связка услуг и специалистов (service_specialists — таблица M:N), контакты клиники (contacts), материалы/статьи (materials), вакансии (vacancies), партнёры (partners). **Контентные данные:** категории вопросов (question_categories), вопросы-ответы (questions), отзывы (feedbacks) с опциональной привязкой к услуге. **Операционные данные:** пользователи-пациенты (patients) с полем роли, письма главному врачу (letters) и сообщения в переписке (letter_messages), чаты с операторами (operator_chats) и сообщения в чатах (operator_chat_messages), записи на приём (appointments). Подробное описание сущностей, связей и атрибутов приведено в документе DATA_MODEL_PROJECT.md.

## 2.2 Процессы

Основные функции приложения реализуются через веб-интерфейс и API (Next.js API Routes), данные хранятся в PostgreSQL и обрабатываются через Prisma. Ключевые бизнес-процессы:

- **Авторизация и аутентификация.** Регистрация нового пациента (логин, email, пароль, ФИО, телефон), вход по учётным данным (NextAuth). Разделение ролей: Пациент (USER), Администратор (ADMIN), Главный врач (CHIEF_DOCTOR), Оператор (OPERATOR). Сессия хранится в cookie, защищённые маршруты проверяют роль и идентификатор пользователя.

- **Каталог и запись на приём.** Просмотр древовидного меню услуг и списка специалистов, карточки услуги (описание, цена, медиа, привязанные специалисты). Запись на приём: выбор специалиста, опционально услуги, дата и время, длительность; сохранение записи в таблицу appointments со статусом (pending, confirmed, cancelled, completed, no_show). Пациент видит свои записи в личном кабинете.

- **Письма главному врачу.** Пациент создаёт письмо (тема, текст); главный врач просматривает входящие, отвечает (поле reply и переписка в letter_messages). Поддерживаются флаги: письмо прочитано врачом, ответ прочитан пациентом, есть новое сообщение от пациента. Переписка — цепочка сообщений с указанием отправителя (patient / chief_doctor).

- **Чаты с оператором.** Пациент создаёт чат; чат попадает в очередь (статус WAITING). Оператор назначает себя на чат (operator_id), статус переходит в ACTIVE. Обмен сообщениями (operator_chat_messages), учёт непрочитанных для оператора и пациента. Закрытие чата (CLOSED). В админ-панели — список чатов с фильтрацией и назначением оператора.

- **Администрирование.** Управление справочниками: категории услуг (включая дерево), услуги, специалисты, привязка специалистов к услугам (M:N). Управление контентом: вопросы и категории вопросов, отзывы (в т.ч. верификация), материалы, вакансии, партнёры, контакты. Модерация: просмотр и ответы на письма (главный врач), ведение чатов (оператор). Управление пользователями: блокировка отправки сообщений (is_messages_blocked). Роли ADMIN и CHIEF_DOCTOR имеют доступ к разделам писем и отзывов; оператор — к чатам и назначению себя на чаты.

- **Личный кабинет пациента.** Просмотр и редактирование профиля (ФИО, телефон, аватар), история записей на приём, список писем главному врачу с возможностью открыть переписку, список чатов с операторами и обмен сообщениями.

## 2.3 Описание внешнего пользовательского интерфейса

Интерфейс реализован как многостраничное веб-приложение на Next.js (App Router) с адаптивной вёрсткой. Используются компоненты MUI (Material UI), Radix UI и стилизация Tailwind CSS.

**Публичная часть (без входа).** Главная страница с общим представлением о клинике. Каталог услуг — древовидное меню (service_categories), при выборе категории — список услуг и специалистов; отдельные страницы услуги и специалиста с детальной информацией и медиа. Разделы: Контакты (адрес, карта, часы работы, телефоны, email), Материалы (статьи с фильтром по году), Вакансии, Партнёры. Блок вопросов-ответов (по категориям вопросов) и отзывов. Форма записи на приём: выбор специалиста (и опционально услуги), календарь/время, отправка записи (после этого возможна подсказка о необходимости входа для просмотра в ЛК).

**Личный кабинет пациента (после авторизации).** Навигация по разделам: Профиль (учётные данные, аватар), Письма главному врачу (список писем, создание нового, просмотр переписки), Чаты с оператором (список чатов, создание чата, окно переписки), Записи на приём (список записей с датой, специалистом, статусом). Элементы управления — кнопки «Написать письмо», «Новый чат», «Записаться на приём» с переходом на соответствующие формы.

**Админ-панель (доступ по ролям ADMIN, CHIEF_DOCTOR, OPERATOR).** Общая навигация по разделам: Письма, Чаты, Отзывы, Услуги, Специалисты, Материалы и др. (состав зависит от роли). В каждом разделе — табличный список с колонками, фильтрами и поиском; кнопки «Добавить», «Редактировать», «Удалить» (где разрешено). Для писем — открытие переписки и форма ответа; для чатов — назначение оператора на чат и просмотр сообщений. Модальные окна используются для просмотра карточки, редактирования одной записи и подтверждения удаления. Загрузка медиа (изображения услуг, специалистов, материалов) выполняется через API с сохранением URL в БД (например Cloudinary).

## 2.4 Средства защиты и разграничения доступа к данным

Доступ к приложению обеспечивается обязательной авторизацией: защищённые маршруты (личный кабинет, админ-панель) проверяют наличие сессии NextAuth; при отсутствии сессии выполняется перенаправление на страницу входа. Пароли не хранятся в открытом виде: при регистрации и смене пароля применяется хеширование bcrypt перед записью в поле password таблицы patients.

Разграничение доступа строится на ролях (поле role в patients). **Пациент (USER):** доступ к публичным страницам и к личному кабинету только к своим данным (письма, чаты, записи отфильтрованы по patient_id); доступ к админ-разделам запрещён. **Оператор (OPERATOR):** доступ к разделу чатов, возможность назначать себя на чаты и вести переписку; доступ к другим админ-разделам (услуги, отзывы, письма в полном объёме и т.д.) может быть ограничен. **Главный врач (CHIEF_DOCTOR):** доступ к письмам главному врачу (просмотр, ответ, переписка), к отзывам и, при необходимости, к другим разделам; без полного админ-доступа к пользователям. **Администратор (ADMIN):** полный доступ ко всем разделам админ-панели, включая управление пользователями (блокировка is_messages_blocked). Проверка роли выполняется на сервере при обработке API-запросов и при рендере страниц.

Целостность данных обеспечивается средствами PostgreSQL и Prisma: все связи заданы внешними ключами; при удалении записей применяются политики onDelete (Cascade для зависимых сущностей, например сообщений при удалении чата; SetNull для опциональных связей; Restrict где удаление родителя недопустимо). Это предотвращает появление «висячих» ссылок и согласованность данных при удалении.

## 2.5 Тесты на использование

Функциональное тестирование выполняется по основным сценариям, соответствующим вариантам использования системы. В тест-кейсах проверяются следующие действия.

- **Авторизация:** успешный вход по корректным логину и паролю; неуспешный вход при неверном пароле или несуществующем пользователе; отображение интерфейса в соответствии с ролью (пациент не видит админ-меню, оператор видит раздел чатов и т.д.).

- **Запись на приём:** выбор специалиста и услуги из списка, выбор даты и времени, сохранение записи; появление записи в личном кабинете пациента с корректными данными и статусом; при необходимости — отмена или изменение статуса записи.

- **Письма главному врачу:** создание нового письма пациентом (тема, текст), появление письма в списке входящих у главного врача; ответ главного врача и отображение ответа у пациента; добавление сообщений в переписку (letter_messages), проверка флагов прочтения (is_read, is_reply_read).

- **Чаты с оператором:** создание чата пациентом, появление чата в очереди со статусом WAITING; назначение оператора на чат, смена статуса на ACTIVE; отправка сообщений с обеих сторон, отображение в интерфейсе; закрытие чата, статус CLOSED. Проверка счётчиков непрочитанных сообщений.

- **Админ-функции:** редактирование услуги (название, цена, категория, изображения), сохранение и отображение на сайте; редактирование специалиста и привязка к услугам; модерация отзыва (верификация); блокировка пользователя (is_messages_blocked) и проверка недоступности отправки сообщений заблокированным пользователем.

- **Ограничения доступа:** пациент не может открыть переписку или чат другого пациента (проверка по patient_id на сервере); пользователь с ролью без прав на удаление не получает доступа к действию удаления в интерфейсе и при прямом обращении к API.

Тест-кейсы оформляются в приложении к отчёту (таблица: сценарий, шаги, ожидаемый результат, фактический результат).
